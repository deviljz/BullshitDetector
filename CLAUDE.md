# BullshitDetector 项目规则

## Prompt 工程原则

- **禁止针对特定测试图片写特定提示词**。发现某张图识别结果不对，修复方式应是改进通用规则（如"必须搜索中文译名"），而不是把这张图的答案硬编码进 prompt（如"例：渋谷久留美译为胡桃"这类举例只能用于说明规则，不能成为 prompt 的主要内容）。
- Prompt 改动必须是通用的，适用于所有类似输入，而不只是解决当前这一个案例。
- 下游修复（如 UI 容错显示）只作 fallback，根本问题应在 prompt 或工具层解决。

## 安全原则（最高优先级）

- **任何 API Key、Token、密码、密钥严禁写入代码文件**，只能写入 `config.json`（已在 .gitignore 中）。
- 测试脚本、注释、日志、文档中同样禁止出现真实 Key。
- 提交前必须检查：`git diff --cached` 中不得含任何 key 字符串。

## 自测原则（最高优先级，与安全原则并列）

- **所有 UI 改动必须先自测**：使用 `tests/result_window_test.py` 或同类脚本运行窗口，截图确认布局正确，再让用户验收。
- 自测截图保存到 `tests/` 目录，命名格式：`<功能>_test_<模式>.png`。
- 禁止把明显有布局问题的代码交给用户验收——先自己发现、自己修好。

## 代码质量原则

- 不重复造轮子：多处相似逻辑提取为共用函数，已有 `_tool_loop` / `_run_single` / `_make_card` 等 helper，新功能优先复用。
- 不过度工程化：只做当前需求，不为假设的未来需求提前设计。
- UI 跨线程更新必须通过 `pyqtSignal`，禁止在非主线程直接操作 Qt 组件。
- 并发安全：共享状态（如 `_current_image_b64`）只在单用户桌面场景下使用，不得在并发路径上用实例变量传递请求上下文（见 main.py 的 `captured` dict 模式）。
